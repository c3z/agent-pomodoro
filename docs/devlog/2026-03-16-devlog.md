# 2026-03-16 — Habit Tracker Module (Sprints #32-39)

8 sprintów w jednej sesji. Cały moduł habit trackera od zera do produkcji.

## Dlaczego

Pomodoro śledzi focus time, ale nie śledzi co go wspiera. Ćwiczenia poprawiają focus, sen poprawia completion rate, medytacja redukuje interruptions. Bez śledzenia tych nawyków agent widzi objawy (mniej pomodoro), ale nie przyczyny (brak snu).

Huberman daje naukową strukturę — max 6 nawyków, 21-dniowe cykle formowania, 85% target zamiast perfekcji, 2-day bins zamiast streak counters. Nie "track everything forever", ale świadomy eksperyment z feedbackiem.

## Co zbudowałem

**Backend (Sprint #32):** Dwie nowe tabele Convex, 10 HTTP endpoints, pełny CRUD z checkin/uncheckin, cykl 21-dniowy z auto-advance cronem. Huberman max 6 enforced server-side.

**Agent surface (Sprint #33):** 3 MCP tools dla Claude Desktop, partial name matching w CLI (nie musisz pisać pełnej nazwy), habit-check skill z regułami interpretacji.

**UI (Sprint #34):** Strona /habits z toggle checkboxami, formularz dodawania, cycle indicators. Dashboard widget. Nav link.

**Neuro fidelity (Sprint #35):** 2-day bins — Huberman mierzy nie "ile dni zrobiłeś" ale "ile par dni miało przynajmniej jeden done". 30-day calendar grid per habit.

**Correlation (Sprint #36):** Query który porównuje ile pomodoro robisz w dniach z danym nawykiem vs bez. Agent może powiedzieć "w dniach z exercise masz +40% pomodoro".

**Testing (Sprint #37):** 6 E2E testów. Odkryłem po drodze że zapomniałem zarejestrować route w `app/routes.ts` — strona zwracała 404.

**Docs (Sprint #38):** Aktualizacja 2 skillów — pomodoro-check teraz sprawdza habits na starcie konwersacji, agent-onboarding ma pełną referencję 39 komend i 44 endpoints.

**Integration (Sprint #39):** Weekly retro wzbogacone o habit data — agent widzi korelacje w kontekście tygodniowej retrospektywy.

## 5-agent audit

Odpaliliśmy 5 agentów równolegle (Security, Docs, End-user, Agent Access, Neuroscience Fidelity). Znaleziono i naprawiono:
- IDOR w uncheckin (brak sprawdzania ownership)
- Unbounded .collect() w habitStats (teraz z date range w indeksie)
- getActiveHabits ignorował by_user_active index
- cycleDay duplikowane 5x (wyekstrahowane do computeCycleDay)
- cycleAdvance/cycleAdvanceAll zduplikowane (shared advanceHabitCycles)

Potem 3 agenty /simplify (reuse, quality, efficiency) — złapały to co audit pominął.

## Architektoniczne decyzje

1. **Habits jako osobna tabela** — nie pole na pomodoroSessions. Nawyki żyją niezależnie od sesji.
2. **Date jako string YYYY-MM-DD** — checkin jest per-dzień, nie per-moment. Prostsza logika, prostsze indeksy.
3. **Cron zamiast query-time advance** — cron raz dziennie o 5:00 UTC. Nie obciąża każdego query computation.
4. **POST-only API** — codebase nie używa PATCH/DELETE. Consistency > REST purism.
5. **hubermanTarget w API response** — agent nie musi sam liczyć 85% — dostaje `met: true/false`.

## Czego nie zrobiłem (świadomie)

- **Limbic friction** — nie ma pola difficulty na habitach. Roadmapa mówi #35+ ale to wymaga UX designu.
- **Task bracketing** — start/end timestamps per checkin. Wymaga nowego UX flow, nie tylko checkboxa.
- **Day phase enforcement** — hard/easy jest stored ale nic nie blokuje hard habit o 22:00. Potrzebuje waking time config.
- **verifyUserId DRY** — zduplikowane w 4 modułach. Świadomy dług — refactor to osobny sprint, nie mieszaj z feature work.
